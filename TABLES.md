# Database Tables — Purpose & Frontend Triggers

## Schemas Overview

| Schema | Tables | Purpose |
|--------|--------|---------|
| `app` | 13 tables | Core application data — users, interviews, sessions, chat, documents |
| `interview_bot` | 4 tables | Bot pipeline data — uploaded files, extracted questions, bot sessions |

---

## Schema: `app`

### 1. `app.app_users`

| Field | Detail |
|-------|--------|
| **Purpose** | Stores every registered user — name, email, hashed password, role tag, admin flag, active status, and a counter of completed mock interviews |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `name`, `email`, `password_hash`, `admin`, `active`, `mock_count` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | User fills the Register form → clicks **Create Account** | `/register` | `POST /api/v1/auth/register` |
| 2 | Admin clicks **Promote** on a user row | `/admin/users` | `PATCH /api/v1/users/{id}/promote` |
| 3 | Admin clicks **Deactivate** or **Activate** | `/admin/users` | `PATCH /api/v1/users/{id}/deactivate` or `/activate` |
| 4 | User finishes a mock interview (increments `mock_count`) | `/interviews/:id` | `POST /api/v1/interviews/{id}/finish` |

---

### 2. `app.accounts`

| Field | Detail |
|-------|--------|
| **Purpose** | Stores company / organisation accounts (e.g. PayPal, Ford). Top-level grouping — roles and interview sessions are tagged to an account |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `name`, `logo_url` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | Admin fills **Add New Account** form → clicks **Create Account with Roles** | `/accounts-roles` | `POST /api/v1/admin/accounts-with-roles` |
| 2 | Admin clicks **Edit → Save Changes** on an existing account | `/accounts-roles` | `PATCH /api/v1/accounts/{id}` |
| 3 | Admin clicks **Delete account** | `/accounts-roles` | `DELETE /api/v1/accounts/{id}` |

---

### 3. `app.roles`

| Field | Detail |
|-------|--------|
| **Purpose** | Stores job roles (e.g. Data Engineer, Java Developer) that belong to an account. Users pick a role when starting a mock interview |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `name`, `account_id` (FK → accounts) |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | Admin fills role fields in the **Add New Account** form | `/accounts-roles` | `POST /api/v1/admin/accounts-with-roles` |
| 2 | Admin clicks **Add** inside the **Edit Account** inline panel | `/accounts-roles` | `POST /api/v1/accounts/{id}/roles` |
| 3 | Admin clicks **×** next to a role pill | `/accounts-roles` | `DELETE /api/v1/roles/{id}` |

---

### 4. `app.interview_sessions`

| Field | Detail |
|-------|--------|
| **Purpose** | One row per mock interview attempt. Tracks which user ran it, which account and role it was for, start/end timestamps, and the final AI-generated score and feedback |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `user_id` (FK → app_users), `account_id` (FK → accounts), `role_id` (FK → roles), `started_at`, `completed_at`, `overall_score`, `overall_feedback` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | User selects account + role + experience → clicks **Continue to Mock Interview** | `/interviews` | `POST /api/v1/interviews` — creates the session row and sets `user_id` |
| 2 | User clicks **Finish & Get Report** | `/interviews/:id` | `POST /api/v1/interviews/{id}/finish` — writes `completed_at`, `overall_score`, `overall_feedback` |

---

### 5. `app.questions`

| Field | Detail |
|-------|--------|
| **Purpose** | Stores each interview question generated by the AI for a session. One session has many questions ordered by `order_index` |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `session_id` (FK → interview_sessions), `order_index`, `text` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | Session starts — first question is generated automatically | `/interviews/:id` | First question created inside `POST /api/v1/interviews` |
| 2 | User submits an answer → clicks **Next Question →** | `/interviews/:id` | `POST /api/v1/interviews/{id}/next-question` — AI generates the next question |

---

### 6. `app.answers`

| Field | Detail |
|-------|--------|
| **Purpose** | Stores the candidate's text answer to each question. One answer per question (one-to-one relationship) |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `question_id` (FK → questions, UNIQUE), `text`, `submitted_at` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | User types their answer → clicks **Submit Answer** | `/interviews/:id` | `POST /api/v1/interviews/questions/{questionId}/answer` |

---

### 7. `app.evaluations`

| Field | Detail |
|-------|--------|
| **Purpose** | Stores the AI evaluation of each answer — scores for clarity, depth, quality, and overall, plus written strengths and improvement notes. Created automatically in the same request as the answer |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `answer_id` (FK → answers, UNIQUE), `clarity`, `depth`, `quality`, `overall`, `strengths`, `improvements` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | User clicks **Submit Answer** — Claude evaluates and saves the evaluation immediately after saving the answer | `/interviews/:id` | `POST /api/v1/interviews/questions/{questionId}/answer` |

---

### 8. `app.documents`

| Field | Detail |
|-------|--------|
| **Purpose** | Stores metadata for files uploaded through the document ingestion pipeline — filename, content type, size, extracted text, account/role tags. Used for AI chatbot context retrieval |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `filename`, `content_type`, `size_bytes`, `extracted_text`, `account_name`, `role_name`, `uploaded_at` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | User uploads a single file via the legacy document upload endpoint | `/documents` | `POST /api/v1/documents` |

---

### 9. `app.document_chunks`

| Field | Detail |
|-------|--------|
| **Purpose** | Stores the document split into 1 500-character chunks for keyword / semantic search. One document produces many chunks. Used by the AI chatbot to find relevant context |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `document_id` (FK → documents), `chunk_index`, `text` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | Same upload event as `documents` — chunks are created asynchronously in the background after the document is saved | `/documents` | Created async inside `POST /api/v1/documents` |

---

### 10. `app.chat_sessions`

| Field | Detail |
|-------|--------|
| **Purpose** | Groups a conversation between a user and the AI chatbot. One session holds many messages and tracks when it started and when the last message was sent |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `user_id` (FK → app_users), `started_at`, `last_message_at` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | User sends their first chat message — a new session is created when `sessionId` is null | Chat UI | `POST /api/v1/chat` with `sessionId: null` |

---

### 11. `app.chat_messages`

| Field | Detail |
|-------|--------|
| **Purpose** | Each individual message in a chat session — both the user's question (`role = USER`) and the AI reply (`role = ASSISTANT`) |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `session_id` (FK → chat_sessions), `role`, `content`, `intent`, `created_at` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | User sends a message in the chat — saves the user message and the AI reply as two separate rows | Chat UI | `POST /api/v1/chat` |

---

### 12. `app.real_interview_logs`

| Field | Detail |
|-------|--------|
| **Purpose** | Lets users log real (non-mock) interviews they attended, tagged to an account and panelist name. Used for tracking personal interview history |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `account_id` (FK → accounts), `panelist_name`, `logged_at` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | User fills in the real interview form → clicks submit | `/real-interviews` | `POST /api/v1/real-interviews` |

---

### 13. `app.real_interview_questions`

| Field | Detail |
|-------|--------|
| **Purpose** | Stores the list of questions the user was asked during a real interview. Child rows of `real_interview_logs` — one log has many questions |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `log_id` (FK → real_interview_logs), `text`, `domain`, `service`, `difficulty` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | Same event as `real_interview_logs` — questions are submitted together with the log in a single request | `/real-interviews` | `POST /api/v1/real-interviews` |

---

## Schema: `interview_bot`

### 14. `interview_bot.files`

| Field | Detail |
|-------|--------|
| **Purpose** | Tracks every file uploaded through the Documents page — filename, storage path, uploader ID, and upload timestamp. One row per uploaded file |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `user_id`, `file_name`, `file_path`, `uploaded_at` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | User drags / selects files → clicks **Upload & Process** | `/documents` | `POST /api/v1/process/upload` — saves one `FileRecord` row per file |

---

### 15. `interview_bot.questions`

| Field | Detail |
|-------|--------|
| **Purpose** | Stores interview questions extracted from uploaded files by the LLM. Tagged with company, role, category (SQL / Python / Java / BigQuery / GCS / Others), difficulty, and a `normalized_content` field used for deduplication |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `content`, `normalized_content`, `category`, `company`, `role`, `difficulty` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | User uploads files → the bot pipeline extracts and saves questions | `/documents` | `POST /api/v1/process/upload` → internally calls `POST /api/v1/bot/process/{fileId}` |
| 2 | User starts a bot mock interview — questions are loaded from this table | `/interviews` | `POST /api/v1/bot/interview/start` reads from this table |

---

### 16. `interview_bot.file_question_map`

| Field | Detail |
|-------|--------|
| **Purpose** | Junction table linking each uploaded file to the questions that were extracted from it. Allows tracing which file a question came from and prevents duplicate extraction |
| **Primary Key** | Composite (`file_id`, `question_id`) |
| **Key Columns** | `file_id` (FK → interview_bot.files), `question_id` (FK → interview_bot.questions) |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | Same event as `interview_bot.questions` — rows are inserted automatically when questions are extracted from a file | `/documents` | `POST /api/v1/bot/process/{fileId}` |

---

### 17. `interview_bot.interview_sessions`

| Field | Detail |
|-------|--------|
| **Purpose** | Tracks bot mock interview sessions — which company and role, current status (ACTIVE / COMPLETED), and the user who ran it. Separate from `app.interview_sessions` because the bot uses its own in-memory session engine (Claude API) |
| **Primary Key** | `id` (UUID) |
| **Key Columns** | `user_id`, `company`, `role`, `status`, `created_at` |

| # | Frontend Action | Page | HTTP Call |
|---|----------------|------|-----------|
| 1 | User selects company + role → clicks **Continue to Mock Interview** in the bot section | `/interviews` | `POST /api/v1/bot/interview/start` — creates session with status `ACTIVE` |
| 2 | User clicks **End Interview** | `/interviews` | `POST /api/v1/bot/interview/end` — updates status to `COMPLETED` |

## Quick Reference — All 17 Tables

| # | Table | Schema | Populated when… |
|---|-------|--------|----------------|
| 1 | `app_users` | app | User registers or admin updates a user |
| 2 | `accounts` | app | Admin creates or edits an account |
| 3 | `roles` | app | Admin creates an account with roles or adds a role to an existing account |
| 4 | `interview_sessions` | app | User starts a mock interview (row created); user finishes it (score written) |
| 5 | `questions` | app | Session starts (first question); user clicks Next Question |
| 6 | `answers` | app | User submits an answer |
| 7 | `evaluations` | app | User submits an answer — AI evaluates it immediately in the same request |
| 8 | `documents` | app | User uploads a file through the legacy document endpoint |
| 9 | `document_chunks` | app | Same upload — document is chunked asynchronously in the background |
| 10 | `chat_sessions` | app | User sends their first chat message (new session created) |
| 11 | `chat_messages` | app | User sends a chat message (two rows: user message + AI reply) |
| 12 | `real_interview_logs` | app | User submits a real interview log |
| 13 | `real_interview_questions` | app | Same submission as real_interview_logs |
| 14 | `files` | interview_bot | User uploads files on the Documents page |
| 15 | `questions` | interview_bot | Files are processed — LLM extracts and categorises questions |
| 16 | `file_question_map` | interview_bot | Same processing — maps each file to its extracted questions |
| 17 | `interview_sessions` | interview_bot | User starts a bot interview (ACTIVE); user ends it (COMPLETED) |

## Data Flow Diagram

```
REGISTER          → app.app_users

ACCOUNTS & ROLES PAGE
  Create account  → app.accounts
                  → app.roles
  Add role        → app.roles

INTERVIEWS PAGE
  Start interview → app.interview_sessions  (user_id linked here)
                  → app.questions           (first question generated)
  Next Question   → app.questions           (each subsequent question)
  Submit Answer   → app.answers
                  → app.evaluations         (AI scores the answer instantly)
  Finish          → app.interview_sessions  (overall_score + feedback written)

DOCUMENTS PAGE
  Upload files    → interview_bot.files
                  → interview_bot.questions
                  → interview_bot.file_question_map

BOT INTERVIEW
  Start           → interview_bot.interview_sessions  (status = ACTIVE)
  End             → interview_bot.interview_sessions  (status = COMPLETED)

REAL INTERVIEWS
  Submit log      → app.real_interview_logs
                  → app.real_interview_questions

CHAT
  Send message    → app.chat_sessions       (first message only)
                  → app.chat_messages       (every message, both sides)
```
